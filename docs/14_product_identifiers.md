# Product Identifiers

## Purpose

Core assigns stable identifiers to every sellable `CatalogVariant`.

- SKU is a human-readable internal article number.
- Barcode is the scanner-readable primary internal identifier.

Both belong to `CatalogVariant`, are generated by Core and cannot be changed through normal API operations.

## AQSI constraints

AQSI API V2 accepts a separate `sku` field and a `barcodes` array. Barcode values sent to AQSI must contain only digits and be 4–22 digits long.

Sources:

- <https://api.aqsi.ru/>
- <https://aqsi.ru/support/dobavlenie-tovara/>

## Internal barcode format

Core generates a 13-digit EAN-compatible Restricted Circulation Number:

```text
20 + 10-digit Core sequence + EAN check digit
```

Example:

```text
SKU-000001 -> 2000000000015
```

The `20` prefix identifies a restricted/internal number. It is suitable for use inside the business and can be printed with EAN-13 symbology.

This identifier is not a GTIN and must never be presented as globally unique outside the Core installation.

GS1 references:

- <https://www.gs1ru.org/faq/q-myself-ean13/>
- <https://www.gs1.org/standards/id-keys/company-prefix>

## Ownership rules

- Core generates SKU and internal barcode together from the same reserved sequence number.
- Caller-supplied SKU and internal barcode are rejected.
- Both values are stable for the full lifetime of the Variant.
- Barcode is unique in one Core installation.
- Archived Variants retain their identifiers; values are not reused.
- AQSI initially receives this primary internal barcode.

## Scanner lookup

An exact barcode can resolve its non-archived Variant through:

```text
GET /api/catalog/variants/by-barcode/{barcode}
```

Archived Variants are excluded from normal scanner lookup, but their barcode remains reserved and cannot be reassigned.

## Existing data

The migration preserves existing non-empty barcode values and backfills only missing values. It then requires every Variant to have a unique barcode.

If legacy data contains duplicate barcodes, the migration intentionally stops so the ambiguity can be resolved instead of silently changing an identifier.

## Future multiple barcodes

Manufacturer barcodes, packaging barcodes and additional channel identifiers will require a separate `Barcode` history/entity linked to Variant because one Variant may legitimately have several codes. AQSI already models barcodes as an array.

That extension is intentionally outside the first Ready for Sale implementation. The current `CatalogVariant.barcode` is the required primary internal code.

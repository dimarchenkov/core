# Core Principles

Эти принципы используются для оценки новых функций и архитектурных решений. Исключения должны быть осознанными и зафиксированными в документации.

## 1. Photo First

Первичная фотография является началом регистрации физического объекта.

- Новый товар или арендный объект сначала фотографируют.
- Нельзя откладывать первичную фотографию «на потом».
- Исходное фото сохраняется неизменным.
- Карточка без обязательного фото не может стать готовой к продаже или аренде.

## 2. Process over CRUD

Core строится вокруг рабочих процессов, а не набора таблиц и CRUD-экранов.

Основные режимы: приёмка, подготовка к продаже, продажа, выдача в аренду, возврат, инвентаризация и управление.

SQLAdmin остаётся техническим инструментом, но не определяет пользовательский опыт.

## 3. Physical Objects

Core учитывает не только товарные карточки, но и реальные физические объекты.

- `Product` описывает семейство товара.
- `Variant` является продаваемой складской позицией.
- `Asset` представляет конкретный экземпляр для аренды.

Продажа и аренда используют общую основу, но имеют разные жизненные циклы.

## 4. Every Action Has an Author

Каждое значимое действие должно иметь автора и время выполнения.

Атрибуция нужна для аудита, обучения сотрудников, расследования ошибок и операционной аналитики. Системные операции без пользователя допускаются только явно.

## 5. Everything Is Measurable

Рабочие процессы должны позволять получать метрики без искажения первичной истории.

Core должен в перспективе отвечать на вопросы:

- сколько операций выполнено;
- кто их выполнил;
- сколько времени они заняли;
- сколько было ошибок, отмен и повторных действий;
- какие этапы создают задержки.

Audit Log сохраняет историю, Activity Feed помогает в ежедневной работе, Metrics поддерживает управленческие решения.

## 6. Mobile First

Основные операции должны быть удобны с телефона.

Камера и фотобиблиотека являются штатными источниками изображений. Критические действия не должны требовать настольного компьютера.

## 7. Fast Intake

Регистрация новой позиции с телефона должна занимать не более двух минут.

Повторная поставка существующего варианта должна оформляться быстрее и не требовать повторного создания каталожных данных.

## 8. Zero-to-Working

Путь от пустого поддерживаемого сервера до работающего Core должен быть понятным, воспроизводимым и занимать не более десяти минут при наличии подготовленной конфигурации.

Целевой сценарий:

```text
получить дистрибутив → заполнить конфигурацию → запустить bootstrap → войти в Core
```

Ручные SQL-запросы и неописанные действия не могут быть обязательной частью установки.

## 9. API First

Каждая полноценная бизнес-возможность должна быть доступна через документированный API.

Интерфейсы являются клиентами Core. Пользователь или партнёр может создать собственный web, mobile, kiosk или messenger-интерфейс без переписывания ядра.

Внутренние аварийные и обслуживающие операции могут оставаться только в локальном CLI.

## 10. Documentation as Product

Документация является частью продукта и проверяется вместе с кодом.

Core должен иметь отдельные материалы для:

- конечного пользователя;
- администратора и владельца;
- оператора франшизы;
- разработчика API;
- развёртывания и обновления;
- инвестора или партнёра.

Документация должна объяснять не только «как», но и «зачем».

## 11. Human First

Рабочий процесс оптимизируется под человека и бизнес-результат, а не под структуру базы данных или удобство CRUD.

- Сотрудник видит понятные действия и следующий шаг.
- Повторный ввод и лишние переключения сокращаются.
- Ошибки объясняются человеческим языком и предлагают безопасное исправление.
- Карточки служат справочником, состоянием и историей.
- Ежедневная работа выполняется через workflows.

Mobile First и API First поддерживают Human First, но не заменяют его.
